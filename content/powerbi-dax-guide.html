Power BI DAX & Semantic Modeling

A masterclass reference guide on semantic data modeling, DAX engine optimization, evaluation contexts, filter manipulation, time intelligence, and enterprise-grade Row-Level Security (RLS).

📈 Semantic Data Modeling & Star Schemas

Data Analysis Expressions (DAX) is the functional, query, and formula language that powers the Tabular Engine in Microsoft Power BI. Rather than just applying calculations, DAX operates directly on top of a semantic data model, which should ideally be designed as a Star Schema.

1. Data Modeling: Fact Tables vs. Dimension Tables

Building an efficient BI model requires separating transactional events from descriptive metadata:

  • Fact Tables: Contain quantitative measurements or metrics (e.g., Sales Amount, Quantity, Costs) and foreign keys linking to dimensions. Fact tables are tall, narrow, and grow rapidly.
  • Dimension Tables: Contain descriptive, attributes (e.g., Product Category, Customer Name, Store Region) used to filter, group, and slice facts. Dimension tables are short, wide, and unique.
  • Active vs. Inactive Relationships: A fact table might have multiple date columns (e.g., OrderDate, ShipDate, PaymentDate) linking to the same Date dimension (role-playing dimension). Power BI only allows one active relationship at a time. The active relationship propagates filters automatically, while others remain inactive. 
    -- Activating an inactive relationship dynamically inside a measure
ShipSales = CALCULATE(
    SUM(Sales[sales_usd]),
    USERELATIONSHIP(Sales[ShipDateKey], DateTable[DateKey])
)
  • Cross-Filter Direction: Set relationship cross-filtering to Single by default. Using Both allows bidirectional filtering but causes performance degradation, ambiguity, and circular dependency hazards.

2. Row Context vs. Filter Context

Every DAX expression is evaluated within one or both of the following evaluation contexts:

⚡ The Two Contexts

1. Row Context (Iterators & Calculated Columns): The concept of the "current row." When you create a Calculated Column or use an iterator (e.g., SUMX, FILTER), DAX evaluates the expression row-by-row. Row context does not automatically filter other tables or propagate across relationships.

2. Filter Context (Measures & Visuals): The set of active filters applied to the model. Filters propagate automatically from the 1-side of a relationship to the many-side. It is created by slicers, report filters, columns/rows in a visual, and modified via code.

Context Transition: The conversion of a Row Context into a Filter Context. This happens automatically when you reference a measure inside a row context, or wrap a row calculation inside CALCULATE. 

-- Calculated Column showing Context Transition (row context -> filter context)
-- Without CALCULATE, this would sum the entire table. With CALCULATE, it transforms 
-- the current row attributes into filters, summing only sales for this specific customer.
CustomerTotalSales = CALCULATE(SUM(Sales[sales_usd]))

3. The CALCULATE Function (Filter Modification)

CALCULATE is the absolute gateway to advanced DAX, acting as the only function capable of creating, overriding, or modifying the active Filter Context.

-- 1. Overriding a filter using ALL (Percentage of Total Sales)
PctOfTotalSales = 
VAR CurrentSales = SUM(Sales[sales_usd])
VAR AllSales = CALCULATE(SUM(Sales[sales_usd]), ALL(Sales))
RETURN
    DIVIDE(CurrentSales, AllSales, 0)

-- 2. Intersecting filters instead of overriding using KEEPFILTERS
-- Keepfilters ensures that if the visual slices by 'Category', the category 
-- is intersected with 'Technology' rather than overwritten.
TechSalesKeep = CALCULATE(
    SUM(Sales[sales_usd]),
    KEEPFILTERS(Product[category] = "Technology")
)

4. Time Intelligence Expressions & Date Tables

Time intelligence allows you to calculate year-over-year growth, running totals, and moving averages. To use time intelligence in DAX, you must have a dedicated, marked Date Table with continuous date ranges and no gaps.

-- 1. Year-to-Date (YTD) cumulative total
YTD_Revenue = TOTALYTD(SUM(Sales[sales_usd]), DateTable[Date])

-- 2. Same Period Last Year (YoY Compare)
PriorYearRevenue = CALCULATE(
    SUM(Sales[sales_usd]),
    SAMEPERIODLASTYEAR(DateTable[Date])
)

-- 3. Custom period shifts (e.g., previous month sales)
PrevMonthSales = CALCULATE(
    SUM(Sales[sales_usd]),
    DATEADD(DateTable[Date], -1, MONTH)
)

5. Optimization with Variables (VAR / RETURN)

Variables should be used in all non-trivial measures to improve code readability, simplify debugging, and boost processing performance.

🚀 Performance Benefit of Variables

Variables are evaluated exactly once at the moment of definition. If a variable is referenced multiple times in the return block, the tabular engine uses the cached result instead of executing the underlying query multiple times.

-- Optimized YoY Growth Calculation
YoY_Growth_Pct = 
VAR CurrentSales = SUM(Sales[sales_usd])
VAR LastYearSales = CALCULATE(SUM(Sales[sales_usd]), SAMEPERIODLASTYEAR(DateTable[Date]))
VAR SalesDelta = CurrentSales - LastYearSales
RETURN
    DIVIDE(SalesDelta, LastYearSales, 0)

Note: Variables are evaluated in the context where they are declared, not where they are used.

6. Row-Level Security (RLS)

Row-Level Security restricts data access at the model layer for specific users based on their login credentials. RLS filters propagate through relationships just like normal filters.

  • Static RLS: Hardcoded filter rules applied to a role. For example, a role named "US_Sellers" would have a filter on the Customer table: Customer[Country] = "United States".
  • Dynamic RLS: Filters rows dynamically based on the identity of the logged-in user. This uses security functions:
    • USERNAME(): Returns domain name and username (e.g., DOMAIN\user).
    • USERPRINCIPALNAME(): Returns user email address (e.g., user@company.com). Critical for Power BI Service deployments.
-- Dynamic RLS filter applied to the SalesRepresentative dimension table:
-- Filters the table to rows matching the active logged-in Microsoft Entra ID email.
SalesRepresentative[rep_email] = USERPRINCIPALNAME()
