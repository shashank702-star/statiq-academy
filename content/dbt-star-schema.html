dbt Core Setup & Fact Star Modeling

A reference guide to compiling data warehouses using dbt (Data Build Tool) Core to form optimized dimensional layouts (Star Schemas) inside cloud analytical warehouses (Snowflake, BigQuery, Redshift).

⭐ The Star Schema Architecture

Star Schemas partition models into two main layers:

  • Fact Tables (fact_*): Quantitative transactional measurements (e.g. sales, logs, checkout amounts) containing keys linking to dimensions.
  • Dimension Tables (dim_*): Descriptive context attributes (e.g. customer name, product category, region details).

1. Defining Raw Sources (src.yml)

Always reference raw transactional tables inside a source configuration file to isolate model code from schema changes.

# models/staging/src_stripe.yml
version: 2
sources:
  - name: stripe
    database: raw_warehouse
    schema: stripe_api
    tables:
      - name: charges
      - name: customers

2. Staging Layer Model (stg_stripe_charges.sql)

Cleanse, cast data types, and rename raw transaction fields in lightweight staging models.

-- models/staging/stg_stripe_charges.sql
WITH raw_charges AS (
  SELECT * FROM {{ source('stripe', 'charges') }}
)
SELECT
  id AS charge_id,
  customer_id,
  amount / 100.0 AS amount_usd, -- normalize pennies
  UPPER(currency) AS currency_code,
  created_at AS charge_timestamp
FROM raw_charges;

3. Dimension Model (dim_customers.sql)

Build comprehensive dimensions aggregating context profiles.

-- models/marts/dim_customers.sql
WITH customers AS (
  SELECT * FROM {{ ref('stg_stripe_customers') }}
),
charges AS (
  SELECT * FROM {{ ref('stg_stripe_charges') }}
),
customer_spending AS (
  SELECT
    customer_id,
    SUM(amount_usd) AS lifetime_spend,
    COUNT(charge_id) AS total_orders
  FROM charges
  GROUP BY 1
)
SELECT
  c.customer_id,
  c.customer_name,
  c.signup_date,
  COALESCE(s.lifetime_spend, 0.0) AS lifetime_value,
  COALESCE(s.total_orders, 0) AS total_orders
FROM customers c
LEFT JOIN customer_spending s ON c.customer_id = s.customer_id;

4. Testing Schema Assertions (schema.yml)

Verify primary key integrity, null values constraints, and relationship integrity (foreign key mapping) using schema assertions.

# models/marts/schema.yml
version: 2
models:
  - name: dim_customers
    description: "Descriptive master customer records containing lifetime spend metrics"
    columns:
      - name: customer_id
        tests:
          - unique
          - not_null
          
  - name: fact_orders
    description: "Detailed transactional order logs"
    columns:
      - name: order_id
        tests:
          - unique
          - not_null
      - name: customer_id
        tests:
          - not_null
          - relationships:
              to: ref('dim_customers')
              field: customer_id
